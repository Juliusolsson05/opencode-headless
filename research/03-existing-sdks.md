# Existing SDKs — what OpenCode already exposes officially

## Lead

**One canonical SDK exists**: `@opencode-ai/sdk` (TypeScript, MIT, currently
`1.14.39` per `packages/sdk/js/package.json:4-5`, published to npm under the
same release cadence as the `opencode` CLI itself — both are bumped in lockstep
by `script/publish.ts:32-45` against an `OPENCODE_RELEASE`-driven channel).
That's it for SDKs in the language sense — there is **no** Python, Go, Rust,
or Java client. The repo's `sdks/` top-level folder contains a single entry,
`sdks/vscode/`, which is a thin VSCode *extension* (not a programmatic SDK)
that just spawns `opencode --port <random>` in an integrated terminal and
POSTs to `/tui/append-prompt` (`sdks/vscode/src/extension.ts:65,93-100`).

So the "fragmented surface" worry from the brief is mostly unfounded: there
is **one SDK, in one language (TS), with two API generations side-by-side
(`v1` and `v2`)**, both auto-generated from the same `openapi.json`. The TS
package is essentially a *typed HTTP client + a ~100-line process launcher*.

This is materially different from Claude Code / Codex, which expose nothing
comparable. **But** — the SDK is just a *transport* over OpenCode's own HTTP
server (covered by agent 02). Anything the server doesn't expose, the SDK
doesn't expose either. The "75% gap" question (agent 04) does not collapse
just because an SDK exists; it hinges on whether the server is the source of
truth for the TUI, or whether the TUI does work the server can't see.

## What's in `sdks/`

| Entry | Kind | What it actually does |
|---|---|---|
| `sdks/vscode/` | VSCode extension (publisher `sst-dev`, name `opencode`, v1.14.39) | Registers commands `opencode.openTerminal`, `opencode.openNewTerminal`, `opencode.addFilepathToTerminal`. On open: spawns `opencode --port <16384..65535>` in a VSCode terminal, polls `GET /app` until ready, then optionally POSTs `{ text: "@<file>" }` to `/tui/append-prompt`. No imports of `@opencode-ai/sdk`; uses raw `fetch`. |

That's the entire `sdks/` directory. Every other SDK-shaped artifact lives
under `packages/`.

## What's in `packages/sdk/`

```
packages/sdk/
├── openapi.json          # 503 KB, auto-generated, source of truth
└── js/
    ├── package.json      # @opencode-ai/sdk@1.14.39, MIT
    ├── example/example.ts
    ├── script/{build,publish}.ts
    └── src/
        ├── index.ts      # createOpencode() = server + client convenience
        ├── client.ts     # createOpencodeClient() — typed HTTP client
        ├── server.ts     # createOpencodeServer() / createOpencodeTui() — process launchers
        ├── process.ts    # cross-platform stop()/bindAbort() helpers
        ├── gen/          # v1 — generated by @hey-api/openapi-ts (1197 lines sdk.gen.ts)
        └── v2/           # v2 — same generator, different paramsStructure ("flat") (4801 lines)
```

`package.json:11-19` exports six entry points: `.`, `./client`, `./server`,
`./v2`, `./v2/client`, `./v2/server`. Both `v1` and `v2` are shipped in the
same package — v2 is generated with `paramsStructure: "flat"` per
`script/build.ts:32-46`, which means **both surfaces exist concurrently** and
opencode itself imports both shapes (the `Plugin` package at
`packages/plugin/src/index.ts:14` references `@opencode-ai/sdk/v2` for some
types). Pick one for `opencode-headless`; the v2 typings carry richer
parameter shapes (e.g. `directory`/`workspace` ID lifted into the call
signature) and a `Workspace` namespace that v1 lacks.

## SDK surface — entry points (hand-written wrappers)

| Export | Signature | Behavior | Source |
|---|---|---|---|
| `createOpencode(options?)` | `→ { client, server }` | Convenience: starts server, points client at it. | `src/index.ts:8-21` |
| `createOpencodeServer(options?)` | spawns `opencode serve --hostname --port`; parses `opencode server listening on <url>` from stdout; returns `{ url, close() }`. Defaults `127.0.0.1:4096`, 5 s timeout. Config via `OPENCODE_CONFIG_CONTENT` env. | `src/server.ts:22-100` |
| `createOpencodeTui(options?)` | spawns `opencode [--project --model --session --agent]` with `stdio: "inherit"`; returns `{ close() }`. **No I/O capture** — launches the TUI in the parent terminal. | `src/server.ts:102-134` |
| `createOpencodeClient(config?)` | Wraps `@hey-api/client-fetch` with custom `fetch` (no timeout), `x-opencode-directory` → `?directory=` rewriter (v2 also `x-opencode-workspace`), and a `{}`→`Error` interceptor (v2 only). | `src/client.ts:32-55`, `src/v2/client.ts:46-107` |
| `OpencodeClient` (class) | Generated namespaced façade — see below. | `src/gen/sdk.gen.ts:1157-1197` |
| Auth | Local server uses HTTP basic auth via `OPENCODE_SERVER_PASSWORD` env (default user `opencode`). SDK does not manage it — caller passes `auth` through `Config`. Typically unset for local `127.0.0.1`. |

plus one inlined permissions method. Every method returns
`{ data, response, error }` from `@hey-api/client-fetch`; `event.subscribe`
returns an SSE stream typed as `Event` (the union from `types.gen.ts:704-736`).

## v2 `OpencodeClient` namespaces — what's added

`v2/gen/sdk.gen.ts:4659-4801` exposes lazy getters for all v1 namespaces
plus: `worktree`, `question`, `permission` (proper namespace, not the inline
verb), `sync`, `part`, `experimental`, and a nested `v2` namespace. The class
count alone goes from 22 to 41. Per-method counts roughly double. The v2
shape is the one to bind against if we want forward-compat — v1 is still
generated only because something inside opencode consumes the looser
`paramsStructure`.

## Event surface — the union you'd actually subscribe to

The `Event` union (`types.gen.ts:704-736`) is the SSE payload for both
`event.subscribe` (per-directory) and `global.event` (cross-instance, wrapped
in `{ directory, payload }`). Members:

```
EventServerInstanceDisposed, EventInstallationUpdated,
EventInstallationUpdateAvailable, EventLspClientDiagnostics, EventLspUpdated,
EventMessageUpdated, EventMessageRemoved, EventMessagePartUpdated,
EventMessagePartRemoved, EventPermissionUpdated, EventPermissionReplied,
EventSessionStatus, EventSessionIdle, EventSessionCompacted, EventFileEdited,
EventTodoUpdated, EventCommandExecuted, EventSessionCreated,
EventSessionUpdated, EventSessionDeleted, EventSessionDiff, EventSessionError,
EventFileWatcherUpdated, EventVcsBranchUpdated, EventTuiPromptAppend,
EventTuiCommandExecute, EventTuiToastShow, EventPtyCreated, EventPtyUpdated,
EventPtyExited, EventPtyDeleted, EventServerConnected.
```

All 32 are typed-discriminated by `type: "<dotted.name>"`. This is roughly
the OpenCode equivalent of our `SemanticChannel` source set, except it's
authoritative — `proxy | jsonl/screen` doesn't apply because the server *is*
the source of truth and produces these events directly. `EventMessagePartUpdated`
is the streaming-token analogue (carries a `Part` with `text/reasoning/tool/...`
shapes from `types.gen.ts:160-403`), and `EventSessionStatus` /
`EventSessionIdle` are turn-boundary signals (`types.gen.ts:453-480`).

## Stability / versioning / license

- License: **MIT**, set both at `package.json:6` and at the repo root.
- Version: locked to the `opencode` CLI version (`1.14.39` at vendor time),
  bumped together by the same release pipeline (`packages/sdk/js/script/publish.ts`
  reads `Script.channel` from `@opencode-ai/script`).
- Stability: there is **no** documented stability/deprecation policy
  (confirmed via `opencode.ai/docs/sdk` — page exists, says nothing on the
  topic). The fact that they ship `v1` and `v2` simultaneously without
  removing `v1` is the only de-facto signal — they appear willing to
  maintain old surfaces, at least for one major.
- Generation: SDK is fully generated from `openapi.json` via
  `@hey-api/openapi-ts`. Hand-written portion is ~250 LOC across `index.ts`,
  `client.ts`, `server.ts`, `process.ts`. Anything custom (event
  normalization, channel state machine, transcript reattach) has to live
  above the SDK, not inside it.

## Implication for `opencode-headless`

If we adopt the SDK, ~all of `claude-code-headless`'s `proxy/` and
`parsers/Screen*` machinery can be deleted from the OpenCode equivalent —
the SDK + the SSE event union cover the semantic-stream and committed-channel
roles natively. What remains uncertain (and is exactly agent 04's question)
is whether `EventMessagePartUpdated` covers the same surface area as
parsing the actual TUI screen: chrome / overlays / approval dialogs that
might be TUI-only and never hit the event bus. The presence of a `tui.*`
namespace and `EventTui*` events suggests the TUI *is* drivable from the
server side, which is encouraging — but doesn't prove the inverse (that the
server emits everything the TUI shows).

**Recommendation for the synthesis agent**: build `opencode-headless`
directly on `@opencode-ai/sdk/v2`, treat `event.subscribe` as the
`SemanticChannel` source, treat `session.messages` + `session.get` as the
`CommittedChannel` source, and keep the door open to a thin TUI-screen
parser only if agent 04 finds a real gap.
