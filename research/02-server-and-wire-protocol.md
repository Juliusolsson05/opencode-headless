# OpenCode — Server & Wire Protocol

## Punchline

**Yes — `opencode-headless` can consume OpenCode directly over HTTP+SSE
and skip the PTY mirror, the mitmproxy TLS-MITM, and the screen parser
entirely.** OpenCode ships a long-lived, documented, OpenAPI-described
HTTP server (Hono / Effect HttpApi) with `~106` JSON routes plus one
SSE event stream that already carries every semantic event the
existing two packages reconstruct by tailing JSONL and tapping the
provider proxy. The CLI command is a one-liner: `opencode serve
--port=4096 --hostname=127.0.0.1`. The official JS SDK
(`packages/sdk/js`) launches exactly that subprocess and hits the
same surface (`packages/sdk/js/src/server.ts:22-100`).

This is structurally different from Claude/Codex. There is no `messages`-style
proxy to fan out from; the server is the fan-out. Channels still map
cleanly — `SemanticChannel ← /event` SSE, `CommittedChannel ← GET
/session/{id}/message`, `ScreenChannel` is **n/a** for the headless
server backend (the TUI lives in a separate process; the server has
no terminal chrome to parse). Live-owner state machine collapses to
"server is always the live owner."

## SSE event vocabulary (`GET /event`)

Server-Sent Events. Default `event: message`; `data:` is a JSON
envelope `{ id, type, properties }`. SSE serialization in
`packages/opencode/src/server/routes/instance/event.ts:39-89` (Hono
streaming) and `…/httpapi/event.ts:31-65` (Effect Sse encoder). 10-second
`server.heartbeat` keepalives; `server.connected` is sent first; no
explicit terminal event other than `server.instance.disposed`.

```json
{ "id": "01H…", "type": "session.next.text.delta",
  "properties": { "sessionID": "ses_…", "delta": "Hello" } }
```

The `oneOf` schema (`Event` in `openapi.json`) lists **72 variants**.
Defined at:

- Generic registry: `packages/opencode/src/bus/bus-event.ts` and
  `packages/opencode/src/server/routes/instance/httpapi/event.ts`.
- v2 streaming events (the rich semantic stream): `packages/opencode/src/v2/session-event.ts:33-393`.
- Legacy/v1 message events: `packages/opencode/src/session/message-v2.ts:635` (`message.part.delta`) and the `SyncEvent`-derived `message.updated`, `message.removed`, `message.part.updated`, `message.part.removed`.
- Status: `packages/opencode/src/session/status.ts:28-43` (`session.status`, deprecated `session.idle`).
- Permission/question gates: `packages/opencode/src/permission/index.ts:78-79`, `packages/opencode/src/question/index.ts:97-99`.
- File/LSP/MCP/PTY/installation/IDE/worktree/workspace/VCS/TUI: see `BusEvent.define` grep — every one of these surfaces on `/event` for free.

Streaming families relevant to a turn (v2):

| Family | Events |
|---|---|
| Step | `session.next.step.started` / `.ended` / `.failed` |
| Text | `session.next.text.started` / `.delta` / `.ended` |
| Reasoning | `session.next.reasoning.started` / `.delta` / `.ended` |
| Tool input | `session.next.tool.input.started` / `.delta` / `.ended` |
| Tool exec | `session.next.tool.called` / `.progress` / `.success` / `.failed` |
| Shell | `session.next.shell.started` / `.ended` |
| Compaction | `session.next.compaction.started` / `.delta` / `.ended` |
| Retries | `session.next.retried` |
| Switches | `session.next.agent.switched`, `session.next.model.switched`, `session.next.prompted`, `session.next.synthetic` |

**Turn-done signal**: `session.status` event with
`properties.status.type === "idle"` (`status.ts:73-76`). The
`session.idle` event still fires for back-compat. There is no per-turn
`message.completed` — clients reconcile by watching `session.status`.

Ordering: events are pushed onto an `AsyncQueue` in the order
`Bus.publish` is called inside the agent loop. No sequence numbers.
Reconnect is naive — the SSE endpoint replays nothing; durable history
is fetched via `GET /session/{id}/message` after reconnect.

## Lifecycle over the wire

1. **Boot** — caller spawns `opencode serve --port=4096
   --hostname=127.0.0.1`, parses `opencode server listening on
   http://…` from stdout (the SDK's contract,
   `packages/sdk/js/src/server.ts:51-69`). Set `OPENCODE_SERVER_PASSWORD`
   if remote.
2. **Open SSE** — `GET /event` with `Accept: text/event-stream` plus
   `x-opencode-directory: <abs cwd>`. First frame is `server.connected`.
3. **Create session** — `POST /session` (or `POST /api/session/...` for
   v2). Returns `Session.Info` with `sessionID`.
4. **Send prompt** — `POST /session/{id}/message` (v1, chunked JSON
   response with the final assistant message) or `POST
   /session/{id}/prompt_async` / `POST /api/session/{id}/prompt` (v2,
   immediate ack). Live tokens stream on `/event`.
5. **Detect done** — wait for `session.status { sessionID, status: {
   type: "idle" } }` on `/event`, *or* `POST /api/session/{id}/wait`.
6. **Mid-turn approvals** — `permission.asked` / `question.asked` on
   `/event`; reply with `POST /permission/{id}/reply` or
   `/question/{id}/reply|reject`.
7. **Resume** — sessions are server-resident and persistent. Reattach
   = re-open `/event` + paginate `/session/{id}/message`.
8. **Abort** — `POST /session/{id}/abort`. Bus emits `session.status →
   idle` once teardown completes.
9. **Close** — `POST /instance/dispose`; SSE clients see
   `server.instance.disposed`, then disconnect.

## Divergence from claude/codex

| Concern | claude/codex | opencode |
|---|---|---|
| Live semantic source | mitmproxy `/v1/messages` SSE / OpenAI Responses chunks | OpenCode's own `/event` SSE — **first-party**, no provider tap |
| Durable transcript source | JSONL tail (write-on-complete for Claude, live for Codex) | `GET /session/{id}/message` (DB-backed) — pull, not tail |
| Live-owner state machine | proxy ↔ jsonl ↔ screen | server-only — collapses |
| Process model | PTY-host the CLI, parse the screen | spawn `opencode serve`, talk HTTP. No PTY mirror needed for the agent loop. |
| Auth | OAuth tokens reused from CLI cache | HTTP Basic password env var |
| Project selection | per-process cwd | per-request `x-opencode-directory` header (one server can serve many projects) |
| Multi-session | one CLI = one session | one server = many sessions concurrently |

## Gaps / unknowns

- **SSE replay window**: no `Last-Event-ID` handling found. A client that disconnects mid-turn will miss deltas; reconciliation must come from the message-fetch endpoint. Worth instrumenting.
- **`OPENCODE_EXPERIMENTAL_HTTPAPI` default**: backend selection (`packages/opencode/src/server/backend.ts`) was not read in this pass — assume Hono unless the flag is set, but the codepath both backends share is identical for the routes we need (`/event`, `/session/*`, `/api/session/*`).
- **v2 vs v1 prompt**: the v2 `/api/session` surface has the richer `session.next.*` events; the v1 `/session/{id}/message` route is documented as still streaming via `Hono.stream()` but actually returns one final JSON chunk — the live deltas come through `/event` either way. Whether v2 is required to *receive* `session.next.*` events for a v1-created session is not 100% clear from this read.
- **TUI routes** (`/tui/*`) are intra-process IPC for the CLI's own TUI process and are not useful for `opencode-headless`.
